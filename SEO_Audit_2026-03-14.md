# QuanSynd Website SEO Audit (Code + Build-Pipeline Review)

Date: 2026-03-14

Scope reviewed: `Quansynd-Web/` (Vite + React + React Router + build-time prerender/SSG)

This audit is based on the repo implementation (not a live crawl). Where live data is required (rankings, Core Web Vitals field data, index coverage, backlinks), this report calls it out explicitly.

## Executive Summary

**Overall SEO score (out of 10): 8.1 / 10**

What is already strong (as implemented in this repo):
- **Indexability basics are in place**: `public/robots.txt` allows crawling and references a sitemap, and `public/sitemap.xml` exists.
- **Per-route SEO metadata exists** via `src/seo/Seo.tsx` (title, meta description, canonical URL, OG/Twitter tags, robots meta).
- **Build-time prerender (SSG) is implemented** using SSR rendering (`src/entry-server.tsx`) + `scripts/prerender.mjs`, driven by `scripts/seo-routes.mjs`.
- **Structured data exists**: Organization + WebSite JSON-LD site-wide and BlogPosting JSON-LD on blog post pages.

Top issues / gaps (highest impact remaining):
1. **Unknown routes likely return HTTP 200** due to a catch-all rewrite in `vercel.json`. This weakens technical SEO (soft-404 behavior) and can waste crawl budget.
2. **Redirect + canonical policy is not enforced at the edge** (www vs apex, http to https, trailing slash normalization). This can lead to duplicate URLs.
3. **Sitemap is route-list driven** (hardcoded routes + blog slugs). It must be kept in sync; it also omits `lastmod`.
4. **Schema is minimal** and blog schema currently uses a default image and parses human-readable dates (works today, but can be made more robust).

## Methodology (Best-Practice, Repo-Based)

This report checks:
- Crawlability and indexability: robots, sitemap, canonical strategy, status codes, redirects, rendering strategy (SSG/SSR/SPA)
- On-page signals: titles, descriptions, headings, internal linking, image alt text
- Rich results + social: JSON-LD, OpenGraph/Twitter cards
- Performance signals that affect SEO: JS/CSS size, image weights, font strategy (note: this is not a Lighthouse run)

## Subagent Reviews (4 Perspectives)

### Subagent 1 - Technical SEO (Crawlability, Indexing, Rendering, Status Codes)

**What we see implemented**
- `public/robots.txt` allows crawling and includes `Sitemap: https://quansynd.com/sitemap.xml`.
- `public/sitemap.xml` exists and includes core pages plus blog slugs; it is generated by `scripts/generate-sitemap.mjs` from `scripts/seo-routes.mjs`.
- Per-route HTML head tags are server-rendered via `react-helmet-async` during prerender:
  - SSR render entry: `src/entry-server.tsx`
  - Prerender runner: `scripts/prerender.mjs`
- The build script wires this up: `package.json` -> `node scripts/generate-sitemap.mjs && vite build && vite build --ssr ... && node scripts/prerender.mjs`.

**Gaps / risks**
- `vercel.json` uses a catch-all route to `/index.html` after filesystem handling. For unknown paths, this commonly results in **HTTP 200 + a client-rendered NotFound page** ("soft 404").
- No edge-level normalization rules are present (www vs non-www, trailing slash policy). Canonical tags help, but redirects are the stronger signal.
- Route list for sitemap/prerender is manual; new blog posts must update `scripts/seo-routes.mjs` or they will not be prerendered/in the sitemap.

**Recommendations (Priority)**
- P0: Revisit `vercel.json` routing so unknown routes return a true 404 (while preserving deep-link routing for known pages).
- P0: Add redirect rules for:
  - https enforcement
  - apex vs www (choose one canonical)
  - trailing slash normalization (choose one canonical)
- P1: Enhance sitemap with `lastmod` (at least for blog posts).
- P1: Make sitemap/prerender routes data-driven if blog content grows (avoid missing pages).

### Subagent 2 - On-Page SEO (Titles, Descriptions, Headings, Internal Links)

**What we see implemented**
- Each primary route uses `Seo` with unique title/description/canonical path:
  - Home: `src/pages/Index.tsx`
  - About: `src/pages/About.tsx`
  - Services: `src/pages/Services.tsx`
  - Blog list + blog posts: `src/pages/Blog.tsx`, `src/pages/BlogPost.tsx`
  - Contact: `src/pages/Contact.tsx`
  - Quan Bench: `src/pages/quan-bench.tsx`
- Blog list and posts use descriptive headings and images with `alt` attributes.

**Gaps / risks**
- Titles/descriptions are strong for brand + general intent, but can be refined for search intent targeting:
  - Services page can benefit from service-specific supporting sections and internal anchors (e.g., /services#ux-research).
- Internal linking from blog posts to relevant services (and vice versa) appears limited; stronger internal link architecture helps topical authority.

**Recommendations**
- P1: Add more "topic cluster" linking:
  - Blog posts -> relevant service pages
  - Service pages -> relevant blog posts / case studies
- P1: Add breadcrumbs (UI + schema) for `/blog/:slug` pages.
- P2: If you plan to target specific keywords, create a keyword-to-page map and ensure titles/H1/subheads reflect it.

### Subagent 3 - Performance / Core Web Vitals (SEO-Adjacent)

**What we can infer from the current build output**
- JS bundle: `dist/assets/index-*.js` ~ 413 kB
- CSS bundle: `dist/assets/index-*.css` ~ 75 kB
- Blog images are reasonable (largest ~ 135 kB) and blog grid images use `loading="lazy"`.
- Fonts are loaded from Google Fonts (preconnect is present in `index.html`).

**Gaps / risks**
- External font loading can affect LCP/CLS if not tuned (field data needed to confirm).
- A single main JS bundle suggests limited route-level splitting (may be acceptable for a small marketing site, but still worth watching).

**Recommendations**
- P1: Consider self-hosting fonts (or ensure an explicit `font-display: swap` strategy).
- P2: Add route-level code splitting for heavy pages/components if the bundle grows.
- P2: Run Lighthouse + check real-user CWV in Search Console once deployed.

### Subagent 4 - Rich Results, Social Previews, Measurement, Off-Page Readiness

**What we see implemented**
- OpenGraph + Twitter card tags are generated by `src/seo/Seo.tsx` and default to `public/og-default.jpg`.
- JSON-LD exists:
  - Site-wide: Organization + WebSite via `src/seo/schema.ts` (wired in `src/App.tsx`)
  - Blog posts: BlogPosting JSON-LD via `src/pages/BlogPost.tsx`

**Gaps / risks**
- Organization schema is minimal (no `sameAs`, address, contactPoint, social profiles).
- BlogPosting schema:
  - uses a default image instead of per-post images
  - parses human-readable dates via `new Date(dateString)`; this works with current data but is brittle across formats/locales
- No obvious Search Console / Bing verification tags (expected to be added during setup).

**Recommendations**
- P1: Expand Organization schema (add `sameAs`, optional `contactPoint`, a real logo image).
- P1: Improve BlogPosting schema (per-post image, ISO dates, optional `dateModified`).
- P2: Add an RSS feed (`/rss.xml`) for blog discovery (nice-to-have).
- P2: Add GSC/Bing verification once properties are created.
